# 에러로그

> 이 문서는 [[Engineering Thinking for Beginners]] 스킬의 핵심 산출물입니다.

---

## 3단계 에스컬레이션 모델

```
에러 발생 → 판단: "이건 어디서 해결해야 하지?"

┌────────────────────────────────────────┐
│ 레벨 1: 현장 책임자                        │
│ (Claude Code가 직접 해결)                  │
│ 판단 기준: 한 파일, 한 줄 단위              │
│ 예: 오타, import 경로, 변수명 실수           │
└────────────────────────────────────────┘
安 풀리면 ↓
┌────────────────────────────────────────┐
│ 레벨 2: 지휘자                             │
│ (시니어가 분석)                            │
│ 판단 기준: 반복되는 에러, 구조적 문제         │
│ 예: 같은 에러 두 번, 고치면 다른 곳 깨짐       │
└────────────────────────────────────────┘
安 풀리면 ↓
┌────────────────────────────────────────┐
│ 레벨 3: 설계 수정                           │
│ (아키텍처 먼저 고침)                        │
│ 판단 기준: 코드 수준에서 해결 불가            │
│ 예: 데이터 구조 자체가 잘못된 경우             │
└────────────────────────────────────────┘
```

---

## 에러 기록 양식

```markdown
## YYYY-MM-DD HH:MM: [에러 요약]

**레벨:** 1 | 2 | 3

**증상:**
[무슨 에러가 났는지]

**원인:**
[왜 발생했는지]

**해결 방법:**
[어떻게 해결했는지]

**소요 시간:** HH:MM ~ HH:MM (약 X분)

**교훈:**
[나중에 같은 실수 하지 않으려면?]
```

---

## 에러 레벨별 평균 소요 시간

| 레벨 | 평균 소요 시간 | 예시 |
|------|---------------|------|
| 1 | 30초~5분 | 오타, import 빠뜨림 |
| 2 | 10분~30분 | 반복되는 에러, 구조적 문제 |
| 3 | 1시간~ | 데이터 구조 재설계 필요 |

---

## 기억할 점

```
1. 에러는 나쁘지 않다 → 학습 기회
2. 처음 보는 에러는 기록 → 다음엔 30초 만에 해결
3. 2번 반복되면 레벨 UP → 코드 수준에서 안 풀림
4. 설계 문제는 코드를 고치는 것으로는 안 풀림
5. 에러를 두려워하지 마라 → "돌아갈 곳"이 있다
```
