# 위임 레벨

**에이전트에게 어디까지 위임할 것인가는 팀 규모가 아니라 하네스 성숙도와 영속적 HITL 크기로 결정된다.** 솔로 풀스택 개발자도 하네스가 성숙하면 L3까지 갈 수 있고, 큰 팀이라도 하네스가 없으면 L1에 머물러야 한다.

## 두 내재적 축

규모 기반 결정론을 대체하는 두 축:

| 축 | 설명 | 성격 |
|---|------|------|
| **하네스 성숙도** | Evaluator가 얼마나 잘 작동하는가 | 시간 축 (튜닝으로 증가) |
| **영속적 HITL 영역** | 도메인 복잡도·위험이 요구하는 사람 개입 범위 | 영역 축 (외부 요인) |

이 두 축의 교점이 현재 프로젝트의 위임 레벨을 결정한다.

## 네 레벨

| 레벨 | 설명 | HITL 시점 | 전제 조건 |
|------|------|-----------|----------|
| **L1** | 대화형 — 모든 산출물 검토 | 매 Turn | 없음 |
| **L2** | Sprint 단위 위임 | Sprint 종료 시 | Evaluator Layer 1~2 신뢰 |
| **L3** | Feature 단위 위임 | Feature 완료 시 | Layer 3 Rubric 포함 Evaluator 성숙 |
| **L4** | 로드맵 위임 — 완전 자율 | 내부 도구 영역만 | 네 자산 모두 축적 + Rubric 안정화 |

## 하네스 성숙도 측정 — 다섯 관찰 신호

수치가 아니라 **추세**가 기준:

1. **PASS/FAIL 비율 안정화** — 초기 FAIL 급증 → 안정화
2. **FAIL → PASS 전환 시간 단축** — 같은 유형의 실패가 재발하지 않음
3. **반례 카탈로그 성장률 둔화** — 새로운 우회 패턴 발견 빈도 감소
4. **Rubric 개정 빈도 감소** — 평가 기준이 안정화
5. **HITL 개입 시간 단축** — 사람이 확인해야 하는 항목이 줄고 명확해짐

## 레벨 조정의 비대칭 규율

- **올리는 것은 보수적으로** — 레벨 상승은 2~3 Sprint 동안 해당 레벨의 조건을 모의 운영하며 검증
- **내리는 것은 즉각적으로** — FAIL 패턴이 반복되면 Sprint 도중에도 강등

## 프로젝트 내 차등 적용

한 프로젝트 안에서도 피처마다 다른 레벨을 적용:

| 피처 유형 | 권장 레벨 | 이유 |
|-----------|----------|------|
| 내부 도구 / 유틸리티 | L3~L4 | 위험 낮음, 롤백 용이 |
| 핵심 비즈니스 로직 | L2 | 정확성 중요, Rubric 튜닝 필요 |
| 인증/결제/보안 | L1~L2 | 영속적 HITL 영역 넓음 |
| 신규 도메인 | L1 | Constitution·Rubric 미확립 |

## L4에 도달한다는 것의 의미

L4는 "모든 것을 위임한다"가 아니다. **영속적 HITL 영역만 남고**, 그 외는 모두 자동화된 상태. Layer 3 Rubric 채점, Constitution 인터뷰, 반례 카탈로그 업데이트가 사람의 손에 남는다.
