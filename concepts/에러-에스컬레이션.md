# 에러 에스컬레이션 (Error Escalation)

## Definition
에러를 **레벨별로 분류**하여 적절한 담당자(CC→CD→재설계)에게エスカレーション하는 판단 프레임워크.

## 세 단계 레벨링

### 레벨 1: 코드 수준의 실수
**담당자: Claude Code (현장책임자)**

특징: 한 파일, 한 줄의 문제. 오타, import 경로, 변수명 실수.
```bash
# 대응: 그냥 CC에게 맡기고 결과만 확인
"이 에러 고쳐줘"
```
**CC가 30초~몇 분 내에 해결.**

---

### 레벨 2: 설계 관련 에러 (CC에서 2회 반복)
**담당자: Claude Desktop (지휘자)**

신호:
- CC에게 두 번 시켰는데 같은 에러가 반복
- 에러를 고쳤는데 다른 기능이 깨짐
- CC가 "설계를 확인해야 할 것 같습니다"라고 말함

**핵심: 설계 문서를 먼저 수정하고, 코드를 수정한다.**
```
CD에게 에스컬레이션 → 원인 분석 → 설계 문서 업데이트 → CC에게 원인 전달
```

---

### 레벨 3: 설계 자체의 결함
**담당자: Claude Desktop + 재설계**

신호:
- "이 구조로는 원하는 동작이 불가능합니다" (CC 보고)
- 여러 파일을 동시에 수정해야 하는 상황
- 데이터 구조 자체를 바꿔야 하는 상황

예: [[기억하는-개발]] Ch.15의 복합 필터 문제 — 필터 설계 원칙(원본/필터 분리)이 없는 상태에서는 어떤 코드를 짜도 꼬임.

## 핵심 원칙

> **두 번 시켜서 안 되면, 지휘자에게 가라.**

| 상황 | 행동 |
|------|------|
| CC 1회 실패 | 다시CC에게 시킴 (자율 수정 시도) |
| CC 2회 실패 | CD에게 에스컬레이션 |
| CD에서도 안 됨 | 재설계 후 CC에게 복귀 |

## 왜 돌아갈 곳이 중요한가

기억하는 개발 vs 기억 없는 바이브코딩의 차이:
- **기억 없음**: 에러 → 복붙 → 또 에러 → 또 복붙 → ... → 포기
- **기억 있음**: "누구에게 가야 하는가"만 판단 → 막히면 다른 곳으로 이동

> 돌아갈 곳이 있다는 것. 그것이 "기억하는 개발"과 "기억 없는 바이브코딩"의 가장 큰 차이다.

## 에러 기록의 가치

로그/에러로그.md에 기록:
- 에러 증상, 시도한 것, 해결 방법, 에러 레벨
- **레벨 2~3이 자주 나오면 설계 단계에서 더 꼼꼼하게 해야 한다는 신호**

시간이 지나면 나만의 트러블슈팅 데이터베이스가 됨.
