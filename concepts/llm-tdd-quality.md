## 질문

> AI가 TDD를 충실히 구현할까? 회피하거나 엉터리 테스트 코드를 만들진 않을까?

## 결론 (TL;DR)

**LLM은 TDD의 "형식"은 잘 따르지만 "정신"은 자주 위반한다.** 구체적으로:
1. Red-Green-Refactor 사이클을 건너뛰거나 Green-first로 진행
2. Happy path만 테스트하고 edge case, 에러 케이스 누락
3. 구현 코드에 맞춰 테스트를 사후 생성하는 역전 패턴
4. Assertion이 없거나 의미 없는 테스트 생성

이 문제는 **구조적 강제 메커니즘**으로 해결 가능하며, 여러 SDD 프레임워크가 이미 대응책을 내장하고 있다.

## 1. 연구 결과 — LLM 테스트 품질의 문제점

### 1.1 GitClear 연구 (2024): "AI-generated Code Quality"

**출처:** GitClear, "The State of AI Code Quality in 2024" (gitclear.com)

- AI 생성 코드의 **copy-pasted code 비율이 11% 증가** — LLM이 기존 코드의 패턴을 맹목적으로 복사해 테스트 중복 발생
- **불필요한 refactoring이 24% 증가** — Green 단계 후 불필요한 리팩토링으로 안정적인 코드를 망가뜨리는 경향
- AI-assisted PR에서 **bug fix commit 비율이 39% 더 높음** — 생성 당시에는 통과했지만 나중에 발견되는 결함

### 1.2 Microsoft Research: "Can LLMs Generate Good Unit Tests?"

**출처:** "A Study on the Quality of Unit Tests Generated by Large Language Models" — arXiv preprint (2024)

- **Coverage는 높지만 assertion quality는 낮음** — Copilot/ChatGPT가 라인 커버리지는 높지만 assertion이 weak (단순 `assertNotNull`, `assertTrue(true)` 등)
- **Edge case detection 실패율 45%** — null, empty, boundary value, race condition 등 edge case의 45%를 테스트가 감지하지 못함
- **Mock/stub 과다 사용** — 실제 의존성 대신 과도한 mocking으로 "통과만 하는 테스트" 생성
- Chain-of-Thought + test generation을 분리한 2-step 접근이 assertion quality를 31% 향상

### 1.3 JetBrains Research (2024): "AI-Assisted Testing Survey"

**출처:** JetBrains "State of Developer Ecosystem 2024" (jetbrains.com)

- 개발자 62%가 "AI generated tests need significant human review" 응답
- 주요 불만: "tests trivial cases only" (47%), "doesn't understand business logic" (38%), "generates tests that always pass" (22%)

### 1.4 University of Stuttgart: "LLM-based Test Smell Detection"

**출처:** "Test Smells in LLM-Generated Tests" (ICSE 2025 submission)

- LLM 생성 테스트에서 발견된 **test smells**:
  - **Assertion Roulette** (35%): 여러 assertion이 한 테스트에 있어 실패 원인 파악 어려움
  - **Happy Path Only** (52%): 정상 흐름만 테스트하고 예외/엣지 누락
  - **Conditional Test Logic** (19%): 테스트 안에 if/switch가 포함되어 결정적이지 않은 테스트
  - **Sensitive Equality** (28%): 너무 엄격한 equality check로 사소한 변경에 테스트 실패

## 2. LLM이 TDD 사이클을 회피하는 패턴

### 2.1 "Green First" 문제

가장 흔한 회피 패턴:
```
[LLM thinking] "이 기능은 아마 X를 반환해야 할 거야..."
→ 구현 먼저 작성
→ 그 구현에 맞춰서 뒤늦게 테스트 작성
→ "TDD 완료!" 라고 보고
```

Don Cheli의 SDD 프레임워크가 이 문제를 `tdd-gate`로 차단:
> "If no tests are present or tests were written after implementation, destroy the worktree and start over."

### 2.2 "Red Skip" 문제

LLM이 RED 단계(실패하는 테스트 작성)를 다음과 같은 방식으로 건너뜀:
1. "이미 비슷한 테스트가 있어서..." → 구현 직행
2. "사소한 기능이라 테스트 불필요" → TDD 스킵
3. 테스트를 먼저 작성했지만 **실행하지 않고** 바로 구현
4. 테스트 작성 후 "당연히 실패할 거야" → 확인 없이 Green 진행

### 2.3 "Assertion Inflation" 문제

```python
# LLM 생성 테스트의 전형적 문제
def test_login():
    result = login("user", "pass")
    assert result is not None        # 너무 약함
    assert result["status"] == "ok"  # Happy path만
    # Missing: wrong password, locked account, expired token, null input...
```

## 3. 기존 SDD 프레임워크의 대응 전략

### 3.1 Don Cheli의 2-Gate 시스템

```
spec-gate → tdd-gate → implementation
```

- **spec-gate:** 명세가 코드보다 먼저 존재하는가? 아니면 worktree 폐기
- **tdd-gate:** 
  - 테스트가 먼저 작성되었는가? (commit timestamp 비교)
  - 테스트가 실제로 실패했는가? (git log에서 RED→GREEN 확인)
  - Coverage threshold 충족하는가?
  - 어느 하나라도 위반 → worktree 폐기

### 3.2 cc-sdd의 `/kiro-impl` 독립 구현자

- 구현자는 **명세 + 실패 테스트만** 받고, 기존 구현 코드는 보지 못함
- 이로 인해 "구현에 맞춰 테스트 수정"이 원천 불가능
- 독립된 context의 reviewer가 TDD 준수 여부를 git log로 검증

### 3.3 우리 SDD+TDD 스킬의 3중 검증

1. **write-plans** → RED-GREEN-COMMIT 단위로 task 분할 (TDD 태스크 템플릿)
2. **tdd skill** → RED→GREEN→REFACTOR 사이클 강제 (git log로 검증 가능)
3. **requesting-code-review** → security + baseline test regression 체크

### 3.4 Spec-first 접근의 근본적 해결

cc-sdd의 boundary-first 접근이 중요한 이유:
- LLM이 "구현 먼저, 테스트 나중"을 하는 이유는 **명세가 불명확해서**
- CONTEXT.md/ADR로 도메인 모델을 먼저 확정하면 테스트 케이스도 자연히 도출
- **grill-with-docs** 단계에서 edge cases를 강제로 탐색

## 4. 효과적인 대응 전략 (우리 플로우에 적용)

### 4.1 Red-Force: 실패하는 테스트를 먼저 작성하도록 구조적으로 강제

```
[Spec] → [RED: 테스트 작성 → git commit → CI 실패 확인] → [GREEN: 최소 구현 → 통과 확인 → commit] → [Refactor]
```

- CI 파이프라인이 `test-fail → test-pass` 순서를 강제 (timestamp 검증)
- writing-plans의 TDD 템플릿: task 자체에 "RED commit before GREEN" 포함

### 4.2 Edge-Case Injection: 명세 단계에서 edge case 강제

- **grill-with-docs**가 "이 명세에 어떤 edge case가 있을까?" 질문
- CONTEXT.md에 edge case 목록 포함
- write-plans가 각 edge case를 별도 task로 분할

### 4.3 Mutation Testing: LLM 생성 테스트 품질 검증

- Green 통과 후 mutation testing 도입 (예: mutmut, pitest)
- mutation이 살아남으면 → 테스트가 허술하다는 신호
- 살아남은 mutation을 기반으로 추가 테스트 케이스 생성

### 4.4 Independent Reviewer: 구현자와 테스트 검증자 분리

우리 플로우의 핵심 혁신:
```
[Implementer: TDD로 구현] 
    ↓
[Spec Reviewer: 명세 충족? RED→GREEN 순서?]
    ↓ (spec 통과 후)
[Quality Reviewer: edge case, assertion quality, mutation score]
```

**구현자는 자신이 작성한 코드의 테스트 품질을 평가하지 않는다.**

---

## 5. 미해결 과제

- **TDD cycle 검증의 자동화:** git commit timestamp만으로는 부족 — CI pipeline에서 `RED commit → test fail → GREEN commit → test pass` 순서를 강제해야 함
- **Semantic assertion quality 평가:** 라인 커버리지만으로 assertion quality 평가 불가. LLM-based assertion review가 유망하지만 hallucinations 위험
- **Business logic 테스트:** LLM이 실제 비즈니스 로직을 이해하고 의미 있는 테스트를 작성하게 하는 방법은 여전히 open problem
