# 오케스트레이션 (Orchestration)

## Definition
AI 협업에서 **세 가지 역할(지휘자, 현장책임자, 기억저장소)**을 분리하고 각자가 잘하는 일을 맡기며 프로젝트를 완성하는 패턴.

## 세 가지 역할

| 역할 | 도구 | 하는 일 | 비유 |
|------|------|---------|------|
| **지휘자** | Claude Desktop | 설계, 분석, 의사결정, 아키텍처 결정 | 뇌의 계획력 |
| **현장책임자** | Claude Code | 구현, 실행, 디버깅, 코드 작성 | 손의执行力 |
| **기억저장소** | Obsidian | 모든 맥락 영구 저장, 검색, 참조 | 뇌의 기억력 |

## 왜 분리가 중요한가

**"전부 만들어줘"는 나쁜 지시다.**
- 한 번에 모든 파일을 만들면 어디서 뭘 한 건지 파악 어려움
- 에러가 나면 어디서 시작된 건지 찾기 힘듦
- [[decomposition]] 원칙 위반

**올바른 패턴: Phase별 단계적 구현**
```
Phase 1: 기본 구조 + 개발 서버
Phase 2: 핵심 기능 (할일 추가/완료/삭제)
Phase 3: localStorage 연동
Phase 4: UI 다듬기
...
```

## 지휘자(Claude Desktop)의 핵심 가치

> 10분 설계 투자 = 수 시간 디버깅 절약

설계 대화에서 묻할 핵심 세 가지:
1. **데이터 구조** — 어떻게 확장할 수 있는가?
2. **컴포넌트 분리** — 역할별 분리가 가능한가?
3. **구현 순서** — 의존 관계상 무엇을 먼저 해야 하는가?

## 현장책임자(Claude Code)의 자율 루프

에러가 나면 Claude Code가 스스로 발견→수정→빌드→확인하는 루프를 자동 진행:
```
에러 발생 → "이 에러 고쳐줘" → CC가 스스로 고침 → 다시 빌드 → 확인
```
