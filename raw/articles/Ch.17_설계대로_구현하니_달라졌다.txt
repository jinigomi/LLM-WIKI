설계가 있다. 아키텍처 v2, Phase 3~6 체크리스트, Phase 3 작업 지시서. 모두 Obsidian에 저장되어 있다.

이번에는 현장 책임자(Claude Code)에게 악보를 주고 시킨다.

Claude Code를 실행하고, 설계 문서를 전달한다:

Ch.15에서는 "마감일 기능 추가해줘"라고만 했다. 이번에는 어떤 데이터 구조로, 어떤 마이그레이션 전략으로, 어떤 컴포넌트에 추가하는지가 명확하다.

Claude Code가 작업을 시작한다:

확인: 브라우저에서 기존 할일이 정상 표시되는지 먼저 본다. 마이그레이션이 성공했으면 기존 항목에 "없음" 카테고리와 마감일 없음이 깔끔하게 표시된다. 그 다음 새 할일에 마감일을 설정해본다.

Ch.15에서는 이 단계에서 기존 데이터가 undefined로 표시됐다. 이번에는? 처음부터 정상이다. 마이그레이션 로직이 설계에 포함되어 있었으니까.

여기서 Ch.15과의 차이가 극명하게 드러난다.

Ch.15에서 카테고리를 추가할 때는, 데이터 구조를 또 변경해야 했다. 마감일 추가 때 바꾼 구조를 카테고리 때 또 바꾸면서 혼란이 시작됐다.

이번에는? Phase 3에서 category 필드를 이미 넣어뒀다. 데이터 구조를 건드릴 필요가 없다. UI만 추가하면 된다. Claude Code가 할 일이 훨씬 단순하고, 기존 코드를 깨뜨릴 위험도 적다.

이것이 "세 기능을 동시에 고려한 설계"의 효과다.

마지막 문장이 핵심이다. Ch.15에서 겪은 구체적인 버그를 테스트 조건으로 명시한다. Claude Code는 시킨 대로 테스트한다. 설계 원칙(원본/필터 분리)이 지켜졌다면 이 테스트를 통과한다.

Claude Code가 "동작합니다"라고 해도, 내가 직접 브라우저에서 확인한다:

Ch.15에서는 3번부터 문제가 터졌다. 이번에는?

설계 원칙이 지켜졌다면, 모두 정상 동작한다. 원본 데이터(todos)와 필터 결과(filteredTodos)가 분리되어 있기 때문이다. 추가/삭제/체크는 원본에 작동하고, 필터는 변경된 원본을 자동으로 다시 계산한다.

설계가 있어도 에러가 안 나는 것은 아니다. 다만 에러의 성격이 다르다.

Ch.15에서의 에러: "왜 이런 일이 벌어지는지 모르겠다. 어디서부터 봐야 하지?" 설계 기반의 에러: "useFilter 훅의 필터 로직에 문제가 있는 것 같다."

설계가 있으면 에러의 위치를 좁힐 수 있다. 필터가 이상하면 useFilter를 보면 된다. 데이터가 이상하면 useTodos나 useLocalStorage를 보면 된다. 각 파일의 역할이 명확하기 때문이다.

그리고 Claude Code가 못 고치는 에러가 나오면? Ch.14에서 배운 대로 지휘자에게 에스컬레이션한다:

핵심 기능이 모두 동작하면, 마지막으로 UI를 다듬고 배포한다.

배포는 Ch.19에서 다룬다. 여기서는 코드 완성까지만.

Ch.15과 Ch.17에서 같은 세 기능(마감일, 카테고리, 필터)을 구현했다. 결과를 비교해보자.

둘 다 Claude Code가 코드를 짰다. 도구가 달라진 게 아니다. 지휘자(Claude Desktop)와의 10분 대화가 달라진 것이다.

솔직하게 말하자. 10분은 익숙해진 후의 시간이다.

처음에는 Claude Desktop과의 대화가 어색할 수 있다. 뭘 물어봐야 할지 모르겠고, 답변이 길면 어디가 중요한지 판단하기 어렵다.

하지만 이것만 기억하자. 지휘자에게 물어볼 핵심은 세 가지다:

이 세 질문만 던져도 대부분의 설계 문제는 예방된다. Claude Desktop이 나머지는 알아서 분석해줄 것이다.
