2025년 2월 2일, 한 트윗이 전 세계 개발자 커뮤니티를 뒤흔들었다.

OpenAI의 공동 창업자이자 테슬라 AI 총괄을 지낸 안드레이 카파시(Andrej Karpathy)가 X(구 트위터)에 이렇게 썼다.

> "나는 이것을 '바이브코딩(vibe coding)'이라 부르기로 했다. 완전히 분위기에 몸을 맡기고, 기하급수적 성장을 받아들이며, 코드가 존재한다는 사실 자체를 잊어버리는 것."

이 트윗은 450만 회 이상 조회되었고, "바이브코딩"은 Collins 사전이 선정한 2025년 올해의 단어가 되었다. 불과 몇 달 만에 전 세계 개발자의 일상 용어로 자리 잡은 것이다.

카파시가 묘사한 바이브코딩의 모습은 이랬다:

- AI에게 원하는 것을 자연어로 말한다
- AI가 코드를 생성한다
- "모두 수락(Accept All)"을 누른다. 코드의 변경 내역(diff)은 읽지 않는다
- 에러가 나면 에러 메시지를 그대로 복사해서 붙여넣는다. 대부분 고쳐진다
- 코드가 내가 이해할 수 있는 범위를 넘어서 커진다
- 가끔 AI가 버그를 못 고치면, 이리저리 바꿔달라고 해서 문제가 사라질 때까지 반복한다

솔직히, 매력적으로 들린다. 키보드를 거의 치지 않고, 음성으로 지시만 하면서, 하루 만에 앱이 만들어진다. 카파시 본인도 Swift를 한 번도 써본 적 없는데 1시간 만에 iOS 앱을 완성했다고 밝혔다.

그런데 카파시의 트윗에는 중요한 단서가 숨어 있다. 그가 이어서 쓴 문장이다:

> "일회성 주말 프로젝트에는 나쁘지 않다."

**일회성.** **주말 프로젝트.**

카파시는 세계 최고 수준의 AI 엔지니어다. 코드를 몰라서 바이브코딩을 하는 게 아니다. 코드를 너무 잘 알기 때문에, 사소한 프로젝트에 시간을 쏟기 싫어서 AI에게 맡긴 것이다. 에러가 나도 문제의 본질을 파악할 수 있는 사람이 "편하게 해보자"고 한 것과, 코드를 전혀 모르는 사람이 "이렇게 하면 된다"고 믿는 것은 완전히 다른 이야기다.

실제로 카파시 자신도 1년 후인 2026년 2월, 이렇게 회고했다:

> "당시에는 LLM의 능력이 충분하지 않아서, 바이브코딩은 주로 재미있는 일회성 프로젝트, 데모, 탐색용으로 쓸 수 있었다. (...) 오늘날 LLM 에이전트를 통한 프로그래밍은 전문가들의 기본 워크플로우가 되어가고 있지만, 더 많은 감독과 검토가 필요하다."

그리고 바이브코딩이 아닌 "에이전틱 엔지니어링(agentic engineering)"이라는 이름을 제안했다. 코드를 직접 쓰지 않지만, AI를 감독하는 **엔지니어링** 기술이 필요하다는 뜻이다.

## "프롬프트 한 줄로 앱 완성" 뒤에 숨겨진 진실

유튜브를 열어보자. "10분 만에 SaaS 만들기", "코딩 없이 앱 만들기", "AI로 하루 만에 서비스 런칭" 같은 영상이 넘쳐난다. 댓글에는 감탄이 이어진다. "세상 바뀌었다", "코딩 배울 필요 없다", "이제 아이디어만 있으면 된다."

하지만 그 영상들이 보여주지 않는 것이 있다.

**첫째, 카메라가 꺼진 후의 시간이다.**

영상에서는 깔끔하게 10분이지만, 실제로는 며칠에 걸친 시행착오를 편집한 것이다. 에러를 잡는 데 3시간, 원하는 디자인이 안 나와서 프롬프트를 20번 바꾸는 데 2시간, 배포하다가 환경 변수 문제로 반나절을 날린 것은 영상에 담기지 않는다.

**둘째, "작동한다"와 "쓸 수 있다"의 차이다.**

경험 많은 개발자 딜런 비티(Dylan Beattie)는 이 차이를 이렇게 설명했다: 웹 폼에서 데이터를 받아 엑셀에 넣는 코드를 짰다고 하자. "야, 우리 오후 만에 Typeform 경쟁자를 만들었어!"라고 할 수 있을까? 아니다. 한 대의 컴퓨터에서 한 번 작동한 것뿐이다. 인코딩, 국제화, 동시 접속, 인증, 모니터링, 결제, 모바일 대응, 배포... 이 중 어느 것도 고려하지 않았다.

**셋째, 보안 문제다.**

바이브코딩으로 만든 앱들을 분석한 결과, 반복적으로 같은 보안 취약점이 발견된다. Supabase나 Firebase 백엔드가 완전히 열린 채로 배포되어 누구나 모든 사용자 데이터를 볼 수 있는 경우, 프리미엄 기능을 브라우저에서 숫자 하나만 바꾸면 해제할 수 있는 경우, 다른 사용자의 데이터를 삭제하거나 수천 달러의 API 크레딧을 하룻밤에 소진시킬 수 있는 경우가 비일비재하다.

실제로 2025년, 한 사용자가 Cursor로 SaaS 앱을 만들어 공개한 지 며칠 만에 보안 공격을 받았다. AI는 코드를 생성할 때 보안을 체계적으로 고려하지 않는다. 입력값 검증을 빠뜨리고, 에러 메시지에 시스템 정보를 노출하고, 검증되지 않은 오래된 라이브러리를 끌어온다. 이런 실수는 AI가 아무리 빨라도, 만드는 사람이 모르면 잡을 수 없다.

## "더 빨라졌다"는 착각

여기서 한 가지 충격적인 연구 결과를 소개한다.

2025년 7월, 비영리 AI 연구 기관 METR(Model Evaluation & Threat Research)은 매우 엄밀한 연구를 발표했다. 대규모 오픈소스 프로젝트에서 평균 5년 이상 경험이 있는 16명의 개발자를 모집하고, 246개의 실제 작업을 무작위로 배정하여 AI 도구를 사용하는 그룹과 사용하지 않는 그룹으로 나누어 비교했다.

결과는 놀라웠다:

- 작업 시작 전, 개발자들은 AI가 자신을 **24% 더 빠르게** 해줄 것이라고 예측했다
- 작업 완료 후, 개발자들은 AI 덕분에 **20% 더 빨랐다** 고 체감했다
- 하지만 실제 측정 결과, AI를 사용한 그룹이 **19% 더 느렸다**

20% 빨라졌다고 느꼈지만, 실제로는 19% 느려졌다. **인식과 현실 사이에 약 40%포인트의 격차** 가 있었던 것이다.

왜 이런 일이 벌어졌을까? 연구진은 몇 가지 원인을 제시했다:

1. **AI가 만든 코드를 검토하고 수정하는 데 예상보다 많은 시간이 소모** 되었다. 개발자들은 전체 작업 시간의 약 9%를 AI가 생성한 코드를 검토하고 고치는 데 썼다.
2. **코딩 모드와 프롬프팅 모드 사이를 오가는 '컨텍스트 스위칭'** 이 집중력을 깨뜨렸다. 코드를 직접 쓸 때는 한 가지 흐름에 몰입하지만, AI를 쓸 때는 "프롬프트를 어떻게 쓸까 → AI 출력을 확인 → 맞는지 검증 → 다시 프롬프트"를 반복하면서 몰입이 계속 끊겼다.
3. **이 개발자들은 이미 해당 코드베이스를 너무 잘 알고 있어서**, AI가 도울 여지가 적었다. AI가 만들어내는 코드가 프로젝트의 암묵적 규칙이나 관습을 따르지 않아서 되려 손이 더 갔다.

이 연구의 핵심 메시지는 이것이다: **AI가 나를 빠르게 해준다는 '느낌'은 실제 속도와 다를 수 있다.** 그리고 이 격차는 특히, 내가 만드는 것이 무엇인지 이해하지 못할 때 더 커진다.

## 코드가 쏟아지는 경험 vs 코드를 이해하는 경험

바이브코딩의 가장 위험한 부분은 **"뭔가 되는 것 같은 느낌"** 이다.

화면에 코드가 쏟아진다. 터미널에서 서버가 실행된다. 브라우저에 페이지가 뜬다. 이 순간의 쾌감은 진짜다. "나도 개발자인가?" 하는 설렘이 밀려온다.

하지만 다음 중 어느 것에도 답할 수 없다면, 그건 개발이 아니라 **구경** 이다:

- 이 프로젝트의 폴더 구조는 왜 이렇게 되어 있지?
- 이 파일은 어떤 역할을 하지?
- 사용자가 버튼을 누르면 데이터가 어디로 가지?
- 이 에러 메시지는 뭘 말하는 거지?
- 내일 여기에 새로운 기능을 추가하려면 어디를 건드려야 하지?

이 질문들에 답할 수 없는 상태에서 코드를 계속 생성하면, 프로젝트는 점점 통제 불능이 된다. AI는 이전에 만든 코드와 충돌하는 새로운 코드를 만들고, 나는 어디서 문제가 생겼는지 찾을 수 없고, 에러 메시지를 AI에게 던지면 AI가 또 다른 코드를 쏟아내고, 그 코드가 또 다른 문제를 만든다.

이 악순환이 바로 바이브코딩의 전형적인 실패 패턴이다:

```bash
프롬프트 → 코드 생성 → "오, 된다!" → 에러 발생 → 에러 복붙 → 코드 생성 
→ "오, 고쳐졌다!" → 새로운 에러 → 에러 복붙 → 코드 생성 → ... 
→ 프로젝트 복잡도 폭발 → "나는 지금 뭘 만들고 있는 거지?" → 포기
```

이것은 "새로운 튜토리얼 지옥(new tutorial hell)"이라고도 불린다. 과거에는 튜토리얼 영상을 보면서 따라 치기만 하고 개념을 흡수하지 못하는 것이 문제였다. 바이브코딩 시대에는 학습 과정 자체가 건너뛰어진다. 문제를 만나면 AI에게 던지고, 작동하는 코드를 받고, 배포하고, 내일 또 비슷한 문제를 만나면 같은 사이클을 반복한다.

## 그럼 바이브코딩은 쓸모없는가?

아니다. 바이브코딩 자체는 강력한 도구다.

중요한 것은 **어떻게 쓰느냐** 다.

카파시가 1년 후 정리한 것처럼, 바이브코딩은 다음과 같은 상황에서 빛난다:

- 일회성 프로토타입을 빠르게 만들 때
- 결과를 시각적으로 바로 확인할 수 있을 때
- 범위가 명확하게 한두 개 파일로 한정될 때
- 망가져도 다시 만들면 그만인 실험적 프로젝트일 때

반대로, 다음과 같은 상황에서 바이브코딩만으로는 반드시 실패한다:

- 며칠 이상 지속되는 프로젝트
- 여러 파일과 기능이 서로 연결된 복잡한 시스템
- 사용자의 데이터를 다루는 서비스
- 팀으로 함께 작업하는 프로젝트

결국 문제는 바이브코딩 자체가 아니다. **바이브코딩 "만" 하는 것** 이 문제다.

AI가 코드를 쏟아내는 속도는 점점 빨라질 것이다. 하지만 코드가 빨리 나올수록, **"내가 뭘 만들고 있는지"를 기억하는 능력** 이 더 중요해진다. AI는 기억해주지 않는다. 대화가 길어지면 이전 맥락을 잊고, 새 세션을 열면 백지 상태에서 시작한다.

그래서 우리에게는 **기록하는 시스템** 이 필요하다.

이 책은 그 시스템을 만드는 방법을 알려준다.

---

> **이 장의 핵심**
> 
> - 바이브코딩은 2025년 2월 안드레이 카파시가 만든 용어로, AI에게 코드를 맡기고 "분위기"로 개발하는 방식이다
> - 카파시 본인도 "일회성 주말 프로젝트"에 적합하다고 한정했고, 1년 후 "에이전틱 엔지니어링"으로 방향을 수정했다
> - METR의 연구에 따르면, AI를 사용한 개발자는 20% 빨라졌다고 느꼈지만 실제로는 19% 느려졌다
> - 바이브코딩의 핵심 문제는 코딩 실력이 아니라, 내가 뭘 만드는지 모르는 상태에서 코드가 계속 쌓이는 것이다
> - AI는 기억해주지 않는다. 기록하는 시스템이 필요하다
