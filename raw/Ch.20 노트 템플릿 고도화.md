Part 4까지 오면서 우리는 꽤 많은 노트를 만들었다. 프로젝트 개요, 체크리스트, 아키텍처 문서, 세션 로그, 에러 로그, 회고. 처음에는 빈 파일에 자유롭게 적었지만, 프로젝트가 반복되면 **패턴** 이 보이기 시작한다.

"매번 같은 항목을 쓰네." "이 항목은 항상 비워두게 돼." "이건 빼먹어서 나중에 후회했어."

이 피드백이 쌓이면 **나만의 템플릿** 이 만들어진다. 이 장에서는 Part 3~4에서 경험한 것을 바탕으로, 더 정교한 템플릿을 만든다.

---

## 프로젝트 기획 노트 템플릿 v2

Part 3(Ch.9)에서 만든 기본 템플릿을 발전시킨다.

```markdown
# [프로젝트 이름]

## 한줄 요약
> 이 프로젝트가 뭔지, 왜 만드는지를 한 문장으로.

## 프로젝트 목적
- 누구를 위한 것인가
- 어떤 문제를 해결하는가
- 완성되면 어떤 모습인가

## 핵심 기능
1. [기능 1] — 한줄 설명
2. [기능 2] — 한줄 설명
3. [기능 3] — 한줄 설명

## 기술 스택
- 프레임워크: 
- 상태 관리: 
- 데이터 저장: 
- 배포: 
- 선택 이유: (왜 이 조합인지 한줄)

## 미결정 사항
- [ ] [나중에 결정할 것 1]
- [ ] [나중에 결정할 것 2]

## 참고
- 비슷한 서비스: 
- 이전 프로젝트에서 가져올 것:
```

v1과 달라진 점:

- **"한줄 요약"** 추가 — 3개월 뒤에 이 파일을 열었을 때 1초 만에 파악 가능
- **"선택 이유"** 추가 — 기술 스택을 왜 골랐는지 기록. 다음 프로젝트에서 같은 고민을 반복하지 않음
- **"참고"** 추가 — 비슷한 서비스나 이전 프로젝트 참조. 지휘자에게 "이런 느낌으로"라고 설명할 때 유용
- **"미결정 사항"** 을 체크리스트로 — 결정되면 체크하고, 결정 내용을 옆에 기록

## 세션 로그 템플릿 v2

Part 3(Ch.12)에서 세션 로그를 배웠다. 실전을 거치면서 더 다듬어진 버전이다.

```markdown
## [날짜] 세션

### 오늘의 목표
- 시작 전에 "오늘은 이것만 한다"를 적는다

### 완료한 것
- 

### 발생한 문제
- 문제: 
- 원인: 
- 해결: 
- 레벨: (1/2/3)

### 다음에 할 것
- 

### 오늘의 메모
- 느낀 점, 아이디어, 나중에 해볼 것
```

v1과 달라진 점:

- **"오늘의 목표"** 추가 — 세션을 시작할 때 먼저 적는다. 이게 없으면 "이것저것 하다가 뭘 했는지 모르겠다"가 된다
- **"발생한 문제"에 레벨 표기** — Ch.18에서 배운 에러 레벨(1/2/3)을 기록. 나중에 "레벨 2 이상 문제만 모아서 봐줘"라고 할 수 있다
- **"오늘의 메모"** 추가 — 세션 중에 떠오른 아이디어나 느낌을 자유롭게 적는 공간

> 💡 세션 로그의 핵심은 **3분 안에 쓸 수 있어야 한다** 는 것이다. 너무 정교하면 안 쓰게 된다. 빈칸이 있어도 괜찮다. "완료한 것"과 "다음에 할 것"만 적어도 충분하다.

## 에러 로그 템플릿 v2

Ch.18에서 에러 로그를 작성했다. 프로젝트가 쌓이면 에러 로그가 **나만의 트러블슈팅 데이터베이스** 가 된다.

```markdown
## [날짜] — [에러 한줄 요약]

### 레벨
1 / 2 / 3

### 증상
- 어떤 상황에서 무슨 일이 벌어졌는가

### 에러 메시지
```

에러 메시지 전문

```markdown
### 원인
- 왜 이 에러가 발생했는가

### 해결
- 어떻게 고쳤는가

### 교훈
- 다음에 이 문제를 예방하려면 어떻게 해야 하는가

### 관련
- 관련 파일: 
- 관련 Phase: 
- 비슷한 이전 에러: [[에러로그#날짜]]
```

핵심은 **"교훈"** 과 **"관련"** 항목이다. 교훈은 같은 실수를 반복하지 않기 위한 것이고, 관련은 비슷한 에러를 빠르게 찾기 위한 것이다. Obsidian의 `[[링크]]` 기능으로 에러 간 연결이 가능하다.

## 기술 의사결정 기록 (ADR)

프로젝트를 하다 보면 "왜 이 기술을 선택했지?"가 기억나지 않는 순간이 온다. ADR(Architecture Decision Record)은 이런 결정을 기록하는 간단한 형식이다.

```markdown
## ADR-001: 상태 관리 라이브러리 선택

### 날짜
2026-03-01

### 상태
 확정

### 맥락
할일 앱에 필터와 카테고리가 추가되면서 상태 관리가 복잡해졌다.
Redux, Zustand, useState 중 선택이 필요.

### 결정
useState + useReducer 조합으로 진행

### 이유
- 이 규모에서 Redux는 과도하다 (지휘자 의견)
- Zustand는 학습 곡선이 있어서 Part 5에서 다루기엔 무리
- useState + useReducer로 충분하고, 나중에 필요하면 전환 가능

### 결과
필터 상태 관리가 깔끔하게 동작. 
복합 필터(카테고리 + 완료)도 useReducer로 처리 가능했음.
```

이 기록이 쌓이면 다음 프로젝트에서 "지난번에 상태 관리를 어떻게 결정했지?"라고 지휘자에게 물을 때, 지휘자가 이 기록을 읽고 더 정확한 제안을 해준다.

> 💡 ADR은 모든 결정에 쓸 필요 없다. **나중에 "왜 이렇게 했지?"라고 물을 가능성이 있는 결정** 에만 쓰면 된다.

## 자주 쓰는 프롬프트 모음

Claude Desktop이나 Claude Code에게 자주 하는 요청을 모아두면 시간을 아낄 수 있다.

`템플릿/프롬프트-모음.md` 에:

```markdown
# 자주 쓰는 프롬프트

## 세션 시작
"내 Obsidian 볼트의 프로젝트/[프로젝트명]/ 폴더를 전부 읽고,
현재 프로젝트 상태를 요약해줘."

## 설계 요청 (지휘자용)
"[기능]을 추가하려고 해. 
아키텍처.md를 읽고, 기존 구조에 어떻게 통합하면 좋을지 설계해줘.
데이터 구조 변경이 필요하면 마이그레이션 계획도 포함해줘."

## 구현 지시 (현장 책임자용)
"아키텍처.md와 [Phase]-지시서.md를 읽고,
해당 Phase를 구현해줘.
각 단계 끝날 때마다 동작 확인해줘."

## 세션 종료
"지금까지 한 일을 요약해줘.
어떤 파일을 만들었고, 어떤 기능이 완성됐고,
다음에 뭘 해야 하는지."

## 에러 에스컬레이션 (지휘자용)
"[기능] 구현 중에 에러가 반복되고 있어.
증상: [증상]
시도한 것: [시도 내용]
아키텍처.md를 보고 설계 수준에서 원인이 있는지 분석해줘."
```

이 모음을 만들어두면 매번 프롬프트를 처음부터 생각하지 않아도 된다. 복사해서 빈칸만 채우면 된다.

## 템플릿을 Obsidian에 세팅하기

Obsidian에서 템플릿을 사용하는 가장 간단한 방법:

1. 볼트에 `템플릿/` 폴더를 만든다
2. 위의 템플릿들을 각각 파일로 저장한다 - `템플릿/프로젝트-기획.md` - `템플릿/세션-로그.md` - `템플릿/에러-로그.md` - `템플릿/ADR.md` - `템플릿/프롬프트-모음.md`
3. Obsidian 설정 → 코어 플러그인 → "템플릿" 활성화
4. 템플릿 폴더를 `템플릿/` 으로 지정

이후 새 노트를 만들 때 `Ctrl/Cmd + T` 로 템플릿을 삽입할 수 있다.

> 💡 템플릿은 고정된 것이 아니다. 프로젝트를 할 때마다 "이 항목은 필요 없었어", "이 항목이 빠졌어"를 반영해서 계속 다듬어간다. **템플릿은 성장한다.**

> - 템플릿은 고정된 것이 아니라, 프로젝트마다 다듬어가는 것이다
