## AI에게 기억을 외주 주지 마라

Ch.1에서 바이브코딩의 화려한 이면을 봤다. Ch.2에서 AI도 까먹고, 나도 까먹는다는 사실을 확인했다.

그럼 답은 뭘까?

간단하다. **까먹지 않는 곳에 적어두면 된다.**

이것은 새로운 아이디어가 아니다. 인류는 수천 년 전에 같은 문제를 해결했다. 기억의 한계를 극복하기 위해 문자를 발명했다. 계약서, 장부, 항해 일지, 설계 도면 — 모두 "미래의 내가 까먹을 것에 대비한 기록"이다.

바이브코딩에도 같은 원리를 적용하면 된다:

1. 프로젝트에 대한 모든 결정과 맥락을 **문서** 에 기록한다
2. 그 문서를 **AI가 읽을 수 있는 형태** 로 관리한다
3. 새 세션을 시작할 때마다 AI에게 **"이 문서를 읽어"** 라고 하면 된다

이것이 이 책이 제안하는 "기억하는 개발"의 전부다. 복잡하지 않다.

하지만 "그냥 메모장에 적으면 되지 않나?"라고 생각할 수 있다. 되긴 된다. 하지만 메모장에 적은 내용을 AI에게 전달하려면 매번 복사-붙여넣기를 해야 한다. 파일이 여러 개면 일일이 열어서 복사해야 한다. 프로젝트가 커지면 어떤 파일을 AI에게 보여줘야 할지 판단하는 것 자체가 일이 된다.

우리에게 필요한 것은 네 가지다:

- **기록을 체계적으로 관리할 수 있는 도구** → Obsidian
- **기록을 읽고 대화할 수 있는 AI** → Claude Desktop
- **기록을 바탕으로 코드를 실행할 수 있는 AI** → Claude Code
- **이 셋을 자동으로 연결하는 파이프라인** → MCP

## Obsidian + Claude + MCP 구조

이 책의 핵심 시스템은 세 개의 도구와 하나의 연결고리로 이루어진다.

### Obsidian — 프로젝트의 뇌 (장기 기억)

Obsidian은 마크다운 기반의 노트 앱이다. 왜 하필 Obsidian인가?

**첫째, 모든 파일이 내 컴퓨터에 있다.** Notion이나 Google Docs와 달리, Obsidian의 파일은 내 로컬 폴더에 `.md` 파일로 저장된다. 클라우드 의존 없이 완전한 소유권을 가진다. 이것은 MCP를 통해 AI가 직접 읽고 쓸 수 있다는 점에서 결정적이다.

**둘째, 마크다운이다.** AI는 마크다운을 가장 잘 이해한다. 마크다운은 AI의 모국어나 다름없다. 복잡한 형식 변환 없이 AI가 바로 읽고, 바로 활용할 수 있다.

**셋째, 링크와 구조화가 가능하다.** Obsidian의 `[[위키링크]]` 기능으로 문서 간 연결을 만들 수 있다. 기획문서에서 기술스택 문서로, 기술스택 문서에서 API 설계 문서로 자연스럽게 이어진다.

**넷째, 가볍고 빠르다.** Electron 기반이지만 로컬 파일을 직접 다루기 때문에 반응이 빠르다. 수백 개의 노트가 있어도 버벅이지 않는다.

### Claude Desktop — 함께 생각하는 동료 (설계 파트너)

Claude Desktop은 Anthropic이 만든 데스크톱 앱이다. 웹 브라우저의 claude.ai와 기능은 비슷하지만, 결정적인 차이가 있다: **MCP를 지원한다.**

MCP를 통해 Claude Desktop은 Obsidian의 노트를 직접 읽을 수 있다. 즉, "내 기획문서를 읽고 아키텍처를 제안해줘"라고 하면, Claude가 실제로 Obsidian에 저장된 기획문서를 열어보고 그 내용을 바탕으로 대화할 수 있다.

이것이 왜 중요한가? 웹의 Claude에게는 매번 문서 내용을 복사해서 붙여넣어야 한다. 하지만 Claude Desktop + MCP에서는 "저 파일 읽어봐"라는 한마디면 된다. 컨텍스트 윈도우를 문서 내용의 복사-붙여넣기로 낭비하지 않아도 된다.

Claude Desktop의 역할은 **설계 파트너** 다. 코드를 짜는 것이 아니라, 무엇을 어떻게 만들지 함께 생각하는 것이다.

### Claude Code — 손이 빠른 실행자 (구현 담당)

Claude Code는 터미널에서 동작하는 AI 코딩 에이전트다. Claude Desktop이 "설계" 담당이라면, Claude Code는 "구현" 담당이다.

Claude Code의 특별한 점은 **직접 파일을 만들고, 명령어를 실행하고, 에러를 확인하고, 수정하는** 자율적인 루프가 가능하다는 것이다. "이 기능을 구현하고 테스트까지 돌려봐"라고 하면, 코드를 쓰고, 빌드하고, 에러가 나면 스스로 수정하고, 테스트가 통과할 때까지 반복한다.

### MCP — 이 셋을 연결하는 신경망

MCP(Model Context Protocol)는 Anthropic이 만든 프로토콜로, AI가 외부 도구와 통신할 수 있게 해주는 표준이다.

비유하자면 이렇다: Obsidian은 뇌, Claude Desktop과 Claude Code는 손과 입이다. MCP는 이 기관들을 연결하는 **신경** 이다. 신경이 없으면 뇌에 아무리 좋은 정보가 있어도 손과 입이 활용할 수 없다.

MCP 덕분에: - Claude Desktop이 Obsidian의 노트를 읽고 대화할 수 있다 - Claude Code가 Obsidian의 설계 문서를 참조하며 코드를 짤 수 있다 - 작업 결과를 다시 Obsidian에 기록할 수 있다

이 연결이 이 책의 시스템을 "메모장에 적는 것"과 근본적으로 다르게 만드는 핵심이다.

## 기억하는 개발의 4가지 핵심 원칙

Anthropic의 CEO 다리오 아모데이와 엔지니어링 팀은 Claude를 효과적으로 활용하는 방법에 대해 여러 차례 실전 원칙을 공유했다. 이 원칙들을 Obsidian 워크플로우와 결합하면, "기억하는 개발"의 구체적인 실천법이 된다.

### 원칙 1: 먼저 생각하게 하세요 (Architecture First)

다리오 아모데이는 AI가 바로 코드를 짜기보다, 문제를 분석하고 계획을 세우는 단계를 거칠 때 결과물이 비약적으로 좋아진다고 강조한다.

**나쁜 방법:**

```bash
"로그인 기능 만들어줘"
```

**좋은 방법:**

```bash
"로그인 기능을 구현하기 위한 설계안(Architecture)을 먼저 마크다운으로 작성해줘.
인증 방식, 필요한 API 엔드포인트, 데이터베이스 스키마, 프론트엔드 컴포넌트 구조를 포함해."
```

설계안이 마음에 들면, 그때 "그 설계대로 구현을 시작해"라고 지시하는 것이 버그를 줄이는 가장 확실한 방법이다.

**Obsidian과 연동하는 법:**

이 원칙의 힘이 진짜 발휘되는 것은 Obsidian과 결합할 때다. Claude Desktop에게 설계안을 요청하고, 그 결과를 **Obsidian에 저장** 한다. 예를 들어:

```markdown
바이브코딩프로젝트/
  설계/
    로그인-설계.md       ← Claude Desktop이 만든 설계안
    데이터베이스-스키마.md  ← 합의된 DB 구조
```

이 설계 문서가 Obsidian에 존재하면: - 내일 새 세션을 열어도 "설계 문서 읽어봐"라는 한마디로 맥락이 복원된다 - Claude Code에게 구현을 맡길 때 "이 설계 문서대로 만들어"라고 지시할 수 있다 - 설계가 변경되면 문서를 업데이트하고, AI는 항상 최신 설계를 참조한다

**기록하지 않은 설계는 존재하지 않는 것과 같다.** AI의 대화창에만 남아있는 설계는 세션이 끝나면 증발한다. Obsidian에 저장된 설계는 영구적이다.

---

### 원칙 2: 컨텍스트의 '신선도' 유지 (The Milk Rule)

Anthropic 엔지니어들은 AI 컨텍스트를 **"우유"** 에 비유한다. 우유는 신선할 때 가장 좋고, 오래되면 상한다. AI의 컨텍스트도 마찬가지다. 오래된 대화가 섞일수록 AI의 집중력이 떨어지고, 엉뚱한 답을 할 확률이 높아진다.

이것은 Ch.2에서 다룬 "컨텍스트 부패(Context Rot)"와 정확히 같은 현상이다. 과학적으로도 증명된 사실을 Anthropic 내부에서는 "우유 규칙"이라는 실용적인 이름으로 부르는 것이다.

**실전 팁:** - 하나의 작업(예: 로그인 버그 수정)이 끝나면 반드시 새 세션을 시작한다 - 첫 번째 대화에서 수정할 파일들을 찾고, 두 번째 대화(새 세션)에서 해당 파일들만 타겟팅하여 수정을 지시한다 - Claude Code에서는 `/clear` 명령어로 컨텍스트를 초기화할 수 있다

**Obsidian과 연동하는 법:**

"우유 규칙"을 따르려면 **자주 세션을 끊어야** 한다. 하지만 세션을 끊으면 맥락을 잃는다. 이 모순을 해결하는 것이 바로 Obsidian이다.

핵심은 **"개발 세션 로그"** 를 남기는 것이다:

```markdown
## 개발 세션 로그 — 2026-02-28 오후

### 이번 세션에서 한 일
- 로그인 API 엔드포인트 구현 완료 (/api/auth/login)
- JWT 토큰 발급 로직 추가
- 비밀번호 해싱에 bcrypt 적용

### 발생한 문제
- CORS 에러 발생 → next.config.js에 도메인 추가로 해결

### 다음 세션에서 할 일
- 프론트엔드 로그인 폼 구현
- 토큰 만료 처리 로직 추가

### 현재 프로젝트 상태
- 백엔드 인증: 80% 완료
- 프론트엔드 인증: 미착수
```

이 로그가 Obsidian에 있으면, 새 세션에서 AI에게 "오늘의 개발 세션 로그를 읽고, 다음 할 일부터 시작해"라고 할 수 있다. 우유는 버리되, 기억은 남기는 것이다.

**세션을 끊는 것이 두려운 이유는 맥락을 잃을까봐다. Obsidian에 기록하면, 세션을 끊는 것이 자유로워진다.**

---

### 원칙 3: 문제를 원자 단위로 쪼개기 (Decomposition)

Claude Code는 복잡한 작업도 수행할 수 있지만, 한 번에 너무 큰 요청을 하면 길을 잃기 쉽다. "전체 쇼핑몰 서비스 만들어줘"라고 하면 AI는 어디서부터 시작해야 할지 판단하기 어렵고, 결과물의 품질이 떨어진다.

**원자 단위(Atomic tasks)** 로 나누어 지시하는 것이 핵심이다:

```bash
❌ "쇼핑몰 만들어줘"

✅ 1단계: "데이터베이스 스키마를 설계해줘"
✅ 2단계: "설계된 스키마에 맞춰 상품 목록 API 엔드포인트 하나만 구현해줘"
✅ 3단계: "구현된 API에 대한 테스트 코드를 작성하고 실행해줘"
```

**Obsidian과 연동하는 법:**

Obsidian에서 **체크리스트 기반의 작업 분해 문서** 를 만든다:

```markdown
## 쇼핑몰 프로젝트 — 구현 체크리스트

### Phase 1: 데이터베이스
- [x] 상품 테이블 스키마 설계
- [x] 사용자 테이블 스키마 설계
- [ ] 주문 테이블 스키마 설계
- [ ] 마이그레이션 스크립트 작성

### Phase 2: 백엔드 API
- [ ] GET /api/products — 상품 목록 조회
- [ ] GET /api/products/:id — 상품 상세 조회
- [ ] POST /api/orders — 주문 생성
- [ ] 각 API에 대한 테스트 코드

### Phase 3: 프론트엔드
- [ ] 상품 목록 페이지
- [ ] 상품 상세 페이지
- [ ] 장바구니
- [ ] 결제 페이지
```

이 문서가 있으면: - AI에게 "체크리스트를 읽고, 아직 안 된 것 중 다음 할 일을 알려줘"라고 할 수 있다 - 한 번에 하나의 원자 작업만 지시하고, 완료되면 체크를 한다 - 프로젝트 전체 진행률을 한눈에 파악할 수 있다 - 일주일 뒤에 돌아와도 "어디까지 했더라?"를 5초 만에 알 수 있다

**큰 문제를 쪼개는 것은 AI를 위한 것이기도 하지만, 나를 위한 것이기도 하다.** 막막한 프로젝트가 체크리스트로 분해되면, "다음에 뭘 해야 하지?"라는 불안이 사라진다.

---

### 원칙 4: 실행과 검증의 루프 (Loop of Verification)

Claude Code의 진가는 코드를 '쓰는' 것이 아니라, **'실행하고 고치는' 피드백 루프** 에 있다.

**나쁜 방법:**

```bash
"로그인 기능 코드를 수정해줘"
```

**좋은 방법:**

```bash
"로그인 기능 코드를 수정하고, 직접 빌드해서 에러가 없는지 확인한 뒤,
테스트까지 통과시켜줘"
```

이렇게 복합 명령을 내리면 Claude Code는 스스로 터미널 명령어를 실행하고, 빌드 에러를 확인하고, 테스트 실패 원인을 파악하고, 다시 수정하는 자율적인 루프를 돌린다. 사람이 중간에 개입할 필요 없이 "코드 수정 → 빌드 → 테스트 → 실패 → 재수정 → 재빌드 → 테스트 통과"까지 한 번에 진행된다.

**Obsidian과 연동하는 법:**

검증 루프의 결과를 Obsidian에 기록한다. 특히 **에러와 해결 방법** 을 남기는 것이 중요하다:

```markdown
## 에러 로그

### 2026-02-28: CORS 에러
- **증상:** 프론트엔드에서 API 호출 시 CORS 에러 발생
- **원인:** next.config.js에 API 서버 도메인 미등록
- **해결:** next.config.js의 allowedOrigins에 localhost:8000 추가
- **교훈:** 새 API 서버를 추가할 때마다 CORS 설정 확인 필요
```

이 에러 로그가 쌓이면: - 같은 에러가 다시 발생했을 때 AI에게 "에러 로그를 읽고, 이전에 같은 문제를 어떻게 해결했는지 확인해"라고 할 수 있다 - Ch.2에서 봤던 "같은 버그를 세 번 고치는" 문제가 사라진다 - 프로젝트 특유의 함정들이 문서화되어, 나중에 팀원이 합류하더라도 같은 실수를 반복하지 않는다

## 네 가지 원칙의 순환 구조

이 네 가지 원칙은 독립적인 팁이 아니라, 하나의 **순환 구조** 를 이룬다:

```
┌─────────────────────────────────────────────┐
│                                             │
│    ① 먼저 설계한다 (Architecture First)        │
│         ↓ 설계 문서를 Obsidian에 저장            │
│                                              │
│    ② 세션을 깨끗이 유지한다 (Milk Rule)           │
│         ↓ 세션 로그를 Obsidian에 기록            │
│                                             │
│    ③ 원자 단위로 쪼개서 실행한다 (Decompose)       │
│         ↓ 체크리스트를 Obsidian에서 관리          │
│                                             │
│    ④ 실행하고 검증한다 (Verification Loop)      │
│         ↓ 에러 로그를 Obsidian에 기록            │
│                                             │
│    → 다음 세션: Obsidian의 기록을 읽고 시작        │
│    → ①로 돌아감                               │
│                                             │
└─────────────────────────────────────────────┘
```

매 단계의 결과가 Obsidian에 기록되고, 그 기록이 다음 단계의 입력이 된다. 이것이 "기억하는 개발"의 실체다.

바이브코딩에서 이 시스템이 없으면: - 설계는 대화창에서 증발한다 - 세션이 바뀌면 백지에서 시작한다 - 전체 그림 없이 되는 대로 만든다 - 같은 에러를 반복한다

Obsidian이 있으면: - 설계는 영구적으로 보존된다 - 세션이 바뀌어도 5분 만에 복귀한다 - 체크리스트로 전체 진행률을 파악한다 - 에러는 한 번만 겪고, 기록으로 남긴다

## Obsidian 볼트 구조 미리보기

이 시스템을 적용한 Obsidian 볼트 구조는 이런 모습이 된다:

```yaml
내프로젝트/
  프로젝트-개요.md            ← 목적, 핵심 기능, 타겟 사용자
  설계/
     아키텍처.md              ← 전체 시스템 구조
     데이터베이스-스키마.md     ← DB 설계
     API-설계.md              ← 엔드포인트 목록
  체크리스트.md               ← 원자 작업 목록과 진행 상태
  세션로그/
     2026-02-28-오후.md       ← 오늘의 작업 기록
     2026-02-27-오전.md       ← 어제의 작업 기록
  에러로그.md                 ← 만났던 에러와 해결 방법
  의사결정.md                 ← 왜 이 기술을, 왜 이 구조를 선택했는지
```

각 파일의 역할과 작성법은 Part 2~3에서 상세히 다룬다. 지금은 이 구조의 존재만 기억하자.

## 이 시스템의 본질

정리하면, "기억하는 개발"은 세 가지 질문에 항상 답할 수 있는 상태를 유지하는 것이다:

1. **나는 지금 무엇을 만들고 있는가?** → 프로젝트 개요 + 설계 문서
2. **어디까지 했고, 다음에 뭘 해야 하는가?** → 체크리스트 + 세션 로그
3. **이전에 어떤 문제가 있었고, 어떻게 해결했는가?** → 에러 로그 + 의사결정 기록

이 세 질문에 답할 수 있다면, 3일 뒤에 돌아와도, 일주일 뒤에 돌아와도, 한 달 뒤에 돌아와도 5분 만에 프로젝트에 복귀할 수 있다. AI에게도, 나 자신에게도.

**바이브코딩이 실패하는 이유는 코딩 실력이 부족해서가 아니다. 기억 시스템이 없어서다.**

이 책의 나머지 부분은 이 시스템을 실제로 구축하고 사용하는 방법을 단계별로 안내한다.

Part 2에서는 Obsidian, Claude Desktop, Claude Code, MCP를 설치하고 연결한다. Part 3에서는 이 도구들로 실제 개발 워크플로우를 진행한다. Part 4에서는 처음부터 끝까지 하나의 프로젝트를 완성한다.

준비됐다면, 이제 도구를 설치하러 가자.

> - 세 가지 질문에 항상 답할 수 있으면 된다: 뭘 만드는가? 어디까지 했는가? 어떤 문제가 있었는가?
