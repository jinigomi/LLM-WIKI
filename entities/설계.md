# Smart Video Speed — Browser Extension

## Phase 1: Framing (문제 정의)

### Q1 — 해결하는 문제
YouTube 라이브에서 속도를 변경하면 버퍼링이 반복되어观看 경험이 최악.
라이브 앞부분(마이너스 재생시간)을 빠르게보고 실시간에 가까워지면 버퍼링 없이 1배속 전환 필요.

**3가지 규칙:**
1. 일반 영상 → 사용자 정의 속도로 항상 재생
2. 라이브 재생 중 → 사용자 속도로 시작 → 버퍼링 전에 자동으로 1배속 전환
3. 라이브 seek → 즉시 사용자 속도로 재생 → 실시간 approached → 버퍼링 전에 1배속 전환

**핵심 메커니즘:** YouTube 라이브는 `playbackRate` 변경 시 버퍼링 발생. `currentTime`이 마이너스 영역에서 0에 수렴할 때 버퍼링 감지 → 1배속 전환 타이밍 조절.

### Q2 — 타겟 사용자
많은 사람에게 배포하고 싶은 제품意图 (완성도 높으면 Chrome Web Store 배포)

### Q3 — 기기/플랫폼
Chrome 계열 (Chrome, Edge, Brave 등 Chromium 기반 브라우저)

### Q4 — 레퍼런스
[GlobalSpeed](https://github.com/polywock/globalSpeed)
- 좋았던 점: 팝업 메뉴, 옵션 토글 자유도, 프리셋, 커스텀 단축키
- 아쉬운 점: 라이브 스트리밍 시 버퍼링 문제

### Q5 — 성공 기준
- 좋은 UI/UX
- 정확하게 작동하는 기능
- 높은 호환성
- 높은 자유도 (설정/옵션)

## Phase 2: Discovery (기술 탐색)

### YouTube 라이브 감지 방법

#### 1. URL 기반 감지
```
- youtube.com/live/{VIDEO_ID}
- youtube.com/watch?v={ID}&live=1
- youtube.com/watch?v={ID}&ab_channel={CHANNEL}(라이브 배지만 있을 때)
```

#### 2. getVideoData() API (IFrame Player)
```javascript
// YouTube IFrame API
player.getVideoData().isLive // boolean — 라이브 여부
player.getVideoData().isLiveDvrEnabled // DVR 활성화 여부
```

#### 3. HTML5 Video currentTime 마이너스 영역
```
YouTube 라이브: currentTime이 음수(-)일 수 있음
- 일반 영상: currentTime >= 0
- 라이브: currentTime ≈ -(실시간 지연 초수)
- 예: -120 → 2분 지연된 실시간
```

#### 4. DOM 기반 감지
```javascript
// YouTube 플레이어 컨테이너에 라이브 표시기
document.querySelector('.ytp-live-badge')
document.querySelector('.ytp-time-display')[contains LIVE]
```

#### 5. playerState 기반
```
YouTube Player State:
1 = BUFFERING (자주 발생하면 라이브 가능성 높음)
3 = PLAYING
```

**핵심 패턴:**

- `chrome.runtime.sendMessage`로 background ↔ content 통신

### 라이브 속도 전환 메커니즘 (핵심 문제)

**문제:**
```
사용자 속도 1.75x 설정 → 라이브 currentTime=-120
→ playbackRate를 1.75로 설정
→ 버퍼링 발생 (YouTube 라이브는 playbackRate 변경 시 버퍼 청크 재요청)
→ 영상이 멈춤/재생 반복
```

**해결 접근:**
```
1. currentTime 모니터링 (requestAnimationFrame 또는 timeupdate)
2. currentTime이 -threshold 이하: 사용자 속도 적용
3. currentTime이 -threshold ~ 0 사이는:
  - 버퍼링 감지 → 즉시 1배속 전환
  - 버퍼링 안 감지 → 사용자 속도 유지
4. currentTime >= 0 (실시간 도달): 1배속 고정
5. 버퍼링 recovered → 사용자 속도로 복귀 (선택적)
```

| 항목 | 가능 여부 | 참고 |
|------|----------|------|
| 라이브 감지 | ✅ | URL, getVideoData().isLive, currentTime 음수 |
| playbackRate 오버라이드 | ✅ | Object.defineProperty 패턴 (GlobalSpeed 검증) |
| 버퍼링 감지 | ✅ | waiting/playing 이벤트 |
| 속도 부드러운 전환 | ⚠️ | 현재 웹 표준은 abrupt만 가능, 사전 전환으로 우회 |
| 실시간 근처 자동 1배속 | ✅ | currentTime 모니터링으로 사전 감지 |

### LiveSync.ts (2026-04-26 신규 생성)

**역할:** 라이브 감지 + 속도 자동 전환

**속도 규칙:**
```
currentTime < 0 (지연 구간) → 사용자 속도 적용
currentTime >= 0 (실시간赶上) → playbackRate 1.0 강제

waiting 이벤트 → 即時 1x切替
playing 이벤트 → 사용자 속도 복귀
```

**주요 로직:**
```typescript
class LiveSync {
  // currentTime 모니터링
  // 라이브 구간 감지
  // 버퍼링 이벤트 수신 → 1배속 전환
  // 버퍼링 해제 → 사용자 속도 복귀
}
```

### MediaTower.ts 수정 (2026-04-26)

**변경 사항:**
- LiveSync 통합
- `gsLiveMode` 플래그 추가 — 라이브 모드 비디오 추적

