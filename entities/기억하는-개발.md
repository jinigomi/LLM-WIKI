# 기억하는 개발 (Remembering Development)

## Definition
AI 협업에서 **모든 맥락을 Obsidian에 영구 저장**하여 세션 간 기억을 유지하는 개발 패러다임. "바이브코딩이 실패하는 이유 = 기록 없음 + 설계 없이 바로 코딩"이라는 통찰에서 출발한다.

## Core Pattern: 기록 → 설계 → 구현 → 기록

```
Obsidian (기록) 
    ↓
Claude Desktop (설계/지휘자) 
    ↓
Claude Code (구현/현장책임자) 
    ↓
Obsidian (기록) 
    → 다시 위로 순환
```

## 왜 작동하는가

| 전통적 바이브코딩 | 기억하는 개발 |
|------------------|--------------|
| 3일 후 "이게 뭐였지?" | 5초 만에 맥락 복구 |
| AI가 이전 대화를 잊어버림 | Obsidian이 영구 기억 |
| 에러 반복 시 매번 헤맴 | 에러 로그로 30초 해결 |
| 의사결정 이유 잊어버림 | 의사결정 로그로 추적 |

## 인간과 AI의 분업

- **인간**: 결정하고 기록하는 역할 — "뭘 만들지", "왜 이렇게 만들었는지"
- **AI**: 제안하고 실행하는 역할 — "어떻게 만들지"

> 코딩 실력이 없어도 된다. 하지만 "뭘 만들고 싶은지 결정하는 능력"과 "결과를 기록하는 습관"은 인간만이 할 수 있는 일이다. 적어도 아직은.

## 한계

1. **복잡한 프로젝트에서는 기록의 양이 방대해짐** — 문서 구조를 체계적으로 관리해야 함
2. **AI가 틀릴 수 있음** — 기록이 있으면 실수를 줄일 수 있지만 완전히 없앨 수는 없음
3. **기록하는 습관이 필요함** — 가장 큰 한계. "이 정도는 기억하겠지"라는 생각이 실패의 시작
